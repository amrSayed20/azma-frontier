# AZMA Research Evidence Program

## Phase 30 Status

This document defines the constitutional evidence-governance process for all future technology evaluations.

It does not evaluate technologies, compare vendors, benchmark products, or recommend adoption.

## 1. Evidence Collection Method

Evidence is collected through a structured cycle:

1. Define evidence question and scope.
2. Capture source metadata before extraction.
3. Extract claim-level evidence with traceable references.
4. Classify evidence by type, confidence, and recency.
5. Store evidence package in governance records.

## 2. Constitutionally Acceptable Evidence Sources

Acceptable sources are:

1. Official technical documentation.
2. Source code repositories and verifiable code inspection.
3. Peer-reviewed or authoritative scientific publications.
4. Independent benchmark publications with transparent methodology.
5. Security advisories, incident histories, and compliance records.
6. Licensing and legal documentation.
7. Operational production validation evidence from credible organizations.
8. Community and maintenance activity signals from verifiable repositories.

Unverifiable claims and marketing-only materials are non-authoritative.

## 3. Evidence Quality Validation

Evidence quality is validated against:

1. Source authenticity.
2. Traceability and citation completeness.
3. Method transparency.
4. Reproducibility potential.
5. Recency and relevance.
6. Internal consistency and contradiction checks.

Evidence failing validation is marked non-qualifying.

## 4. Conflicting Evidence Handling

When evidence conflicts:

1. Record all conflicting claims without suppression.
2. Score each claim by source reliability, recency, and methodological rigor.
3. Escalate unresolved high-impact conflicts to constitutional review.
4. Carry uncertainty explicitly into evaluation confidence.

## 5. Outdated Evidence Retirement

Evidence is retired when:

1. Superseded by newer authoritative evidence.
2. Invalidated by version, policy, security, or licensing changes.
3. Methodology is no longer valid.

Retired evidence remains archived with retirement reason and timestamp.

## 6. Linkage to Research Memory

Every evidence package must be linked to Research Memory with:

1. Evaluation identifier.
2. Source references.
3. Claim lineage.
4. Review outcomes.
5. Version and retirement history.

No formal evaluation may proceed without memory linkage.

## 7. Minimum Evidence Before Formal Evaluation

Minimum required set:

1. Official documentation evidence.
2. Licensing/legal evidence.
3. Security/compliance evidence.
4. Source or implementation-level evidence.
5. Operational maturity evidence.
6. At least one independent validation source.

If minimum evidence is incomplete, evaluation is blocked.

## 8. Evidence Review Workflow

1. Evidence submission.
2. Quality validation.
3. Conflict analysis.
4. Constitutional review decision: accept, conditional accept, reject.
5. Memory registration and publication for evaluation use.

## 9. Evidence Versioning Policy

1. Every evidence package receives a version identifier.
2. Material updates create new versions, not silent overwrites.
3. Version history must preserve change rationale and reviewer identity.
4. Evaluation references must point to exact evidence version.

## 10. Evidence Audit Policy

1. All evidence decisions are auditable.
2. Audit logs must preserve source provenance, review outcomes, and version history.
3. Periodic audit checks verify evidence integrity and retirement correctness.
4. Audit findings must be traceable to constitutional governance records.

## Final Statement

No technology may enter formal evaluation in AZMA OS without compliance with this evidence program.
