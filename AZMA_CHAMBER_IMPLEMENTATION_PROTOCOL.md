# AZMA OS — THE CHAMBER IMPLEMENTATION PROTOCOL
## Permanent Constitutional Law
### Ratified: 2026-07-01

---

## FOUNDATIONAL DECLARATION

From this moment forward, no chamber implementation shall begin with UI.
No chamber implementation shall begin with React.
No chamber implementation shall begin with CSS.
No chamber implementation shall begin with page.tsx.

The architecture always precedes the interface.
The experience always precedes the implementation.

---

## THE THREE CONSTITUTIONAL PHASES

Every chamber must be built through exactly three constitutional phases.
No phase may be skipped.

---

Build everything the user never sees. Including:

- Runtime Contracts
- Services
- Memory Integration
- Operations Bus Integration
- Sovereign Intelligence Integration
- Journey Engine Integration
- Living Companion Integration
- Scheduling
- Events
- Constitutional Articles

- Runtime Bootstrap
- Testing
- Validation
- Production Runtime

No user interface. No React. No CSS. No page.tsx. No visual implementation.

---

## PHASE TWO — THE CHAMBER EXPERIENCE

Stop implementation. Do not write UI. Do not write CSS. Do not write React.

Present the complete Chamber Experience Blueprint. The blueprint must define:

- Why this chamber exists
- The emotional objective
- The chamber personality
- The user's emotional journey
- The first conversation
- The Living Companion behavior
- Motion philosophy
- Visual philosophy
- Audio philosophy
- Transition philosophy
- Relationship with every other chamber
- Constitutional mission

**Implementation is forbidden before approval.**

---

## PHASE THREE — THE FINAL PRODUCTION INTERFACE

Only after constitutional approval... build the complete interface.

This is not a refactor. This is not a modification. This is not a redesign.
This is the first and final production implementation.

Generate:

- page.tsx
- CSS
- Components
- Animations
- Interactions
- Accessibility
- Responsive behavior
- Living Interface

**Everything must be production-ready.**

## NO INCREMENTAL UI

The Chief Architect explicitly prohibits incremental UI modification.

- Never patch existing interfaces
- Never append new UI
- Never partially replace components
- Never evolve the previous chamber visually

Once the experience is approved, the chamber shall be rebuilt as one complete production implementation.
